# DevPilot — 本地代码智能体框架 技术总结

> **定位：** 面向代码仓库长链路任务的本地 Agent 框架  
> **核心命题：** 多轮迭代中的 Context 溢出治理、工具副作用可控、断点恢复零误执行  

---

## Executive Summary

DevPilot 是一个运行在本地的代码智能体框架，专门为解决长链路代码任务中的三个核心痛点而设计：

1. **Context 溢出** — 多轮 ReAct 迭代中 prompt 膨胀导致模型遗忘、幻觉、token 浪费
2. **工具副作用不可控** — 文件误写、重复调用、敏感信息泄露无系统级防护
3. **中断恢复不可靠** — 进程崩溃或上下文超限后无法精确续作，重头执行成本高

通过 **Agent Orchestrator 架构 + 分层记忆缓存 + 断点续作机制 + 五层安全链** 四层设计，DevPilot 实现了：
- Prompt 压缩 **23%**（7,100 → 5,600 tokens），关键信息零丢失
- 重复文件读取从 **60 次降至 0 次**
- 10 种中断场景恢复后 **零误执行**
- 任务成功率、预算内完成率、verifier 通过率均 **100%**

## 1. 项目概述

### 1.1 解决的问题

| 痛点 | 传统方案 | DevPilot 方案 |
|:-----|:---------|:--------------|
| **Context 溢出** | 全量历史拼接，token 线性增长 | 重要性评分裁剪，分层记忆按需注入 |
| **工具副作用** | 依赖 LLM 自觉，无系统兜底 | 五层安全链拦截，高风险操作人工审批 |
| **中断恢复** | 重新执行全部步骤 | 全量快照 + 增量 diff，秒级恢复 |
| **重复 I/O** | Agent 反复读取相同文件 | 分层记忆缓存，文件快照命中率 100% |

### 1.2 核心目标

- **Context Budgeting:** 在预算约束下最大化信息密度，关键信息不丢失
- **Checkpoint & Resume:** 任意节点中断后可精确恢复，不重做已完成的副作用操作
- **Tool Safety:** 所有工具调用经过校验→隔离→审批→去重→脱敏五层保护
- **Tiered Memory:** 记忆按生命周期分级管理，自动回收过期数据

### 1.3 产品定位

本地代码智能体框架 — 面向 IDE 插件、CI/CD 流水线、自动化代码审查等场景。不是通用聊天 Agent，而是**专注于代码仓库长链路任务的工程化框架**，强调可控性、可恢复性、可审计性。

## 4. 核心贡献详解

### 4.1 Agent Orchestrator 架构

| 维度 | 说明 |
|:-----|:-----|
| **范式** | ReAct — Observe → Think → Act 循环，模型决策与工具执行分离 |
| **模型接入层** | 统一抽象接口，支持 2 类 LLM 后端（本地/云端），切换无需改业务逻辑 |
| **工具注册层** | 7 种工具类型：文件读写、代码搜索、Shell 执行、Git 操作、LSP 查询、Diff 生成、测试运行 |
| **状态管理层** | 会话级状态机，追踪当前轮次、已完成步骤、待审批操作、上下文预算消耗 |
| **工件落盘** | 按 `{session_id}/{turn_id}/{artifact_type}` 三维组织，可追溯、可回滚 |

### 4.2 Context Budgeting（上下文预算治理）

| 维度 | 说明 |
|:-----|:-----|
| **核心问题** | 多轮迭代中 ReAct 的 observation 不断追加，prompt 线性膨胀 |
| **解决方案** | 重要性评分 + 裁剪策略：对每轮 observation 计算信息密度分数，低于阈值的压缩为摘要 |
| **评分维度** | 错误信息（高权重）、代码变更内容（中权重）、工具输出（低权重—可摘要替代） |
| **效果** | 12 组场景平均 prompt 从 **7,100 → 5,600 tokens（↓23%）**，关键信息零丢失 |

```
压缩前: 7,100 tokens
压缩后: 5,600 tokens
────────────────────────
节省:   1,500 tokens (23%)
关键信息保留率: 100%
```

### 4.3 分层记忆缓存

| 层级 | 类型 | 生命周期 | 内容 | 淘汰策略 |
|:-----|:-----|:---------|:-----|:---------|
| **L1** | 摘要 | 永久 | 每轮的关键决策、错误教训、文件修改摘要 | 不淘汰（体积小） |
| **L2** | 文件快照 | TTL (可配置) | 已读取文件的缓存副本，跟踪文件哈希 | 时域淘汰 + 文件变更失效 |
| **L3** | 过程笔记 | 会话 | 当前会话中的临时计算结果、中间状态 | 会话结束时回收 |

**核心收益：** follow-up 重复文件读取从 **60 次降至 0 次**。Agent 读取文件前先查 L2 快照 → 哈希比对 → 命中直接返回，避免重复 I/O。

### 4.4 断点续作机制

| 维度 | 说明 |
|:-----|:-----|
| **快照策略** | 全量快照（每轮结束后）：上下文状态、已执行工具列表及副作用、工件目录状态、预算消耗 |
| **恢复策略** | 增量 diff 恢复：对比快照与当前 workspace → 重放未完成步骤 → 跳过已完成副作用 |
| **覆盖场景** | Context 超限 · 进程崩溃 · Workspace 变更 · 网络中断 · API 超时 · 磁盘满 · 权限变更 · 并发冲突 · OOM · 手动暂停 |
| **恢复验证** | 恢复后对比期望状态，确认所有副作用操作已执行且不重复，**零误执行率** |

### 4.5 五层安全链

```
工具调用请求
    │
    ▼
┌──────────────────┐
│ L1: 参数校验      │ 路径合法性、参数类型、范围检查
└──────┬───────────┘
       │ ✓
       ▼
┌──────────────────┐
│ L2: 工作区隔离    │ 所有操作限制在 workspace 内，禁止越界访问
└──────┬───────────┘
       │ ✓
       ▼
┌──────────────────┐
│ L3: 高风险审批    │ rm -rf / force push / 修改 .git/config → 人工确认
└──────┬───────────┘
       │ ✓
       ▼
┌──────────────────┐
│ L4: 重复调用拦截  │ 相同参数的工具调用在 N 秒内 → 返回缓存结果
└──────┬───────────┘
       │ ✓
       ▼
┌──────────────────┐
│ L5: 敏感信息脱敏  │ API Key / Token / 密码 → 自动遮蔽后写入日志
└──────────────────┘
```

**效果：** 任务成功率、预算内完成率、verifier 通过率均 **100%**。

## 5. 评测体系

DevPilot 采用 **四维独立评测** 架构，避免单一分数掩盖模块差异：

### 5.1 评测维度

| 维度 | 评测目标 | 指标 | 方法 |
|:-----|:---------|:-----|:-----|
| **回归测试** | 端到端功能正确性 | 成功率、verifier 通过率 | 固定测试集，验证任务完整执行且输出符合预期 |
| **上下文治理** | Context Budgeting 效果 | Prompt token 数、关键信息保留率 | 对比压缩前后 prompt，验证信息无丢失 |
| **记忆收益** | L2 文件快照命中率 | 重复文件读取次数、I/O 节省量 | 统计同一会话中 follow-up 读文件行为 |
| **恢复正确性** | Checkpoint & Resume 精准度 | 恢复成功次数、误执行次数 | 模拟 10 种中断场景，验证恢复后状态一致性 |

### 5.2 评测路径设计

```
路径 A: 回归测试
  标准任务集 → 全流程执行 → verifier 校验 → 成功率统计
  覆盖: 代码生成、重构、调试、测试编写等典型场景

路径 B: 上下文治理 + 记忆收益
  长链路任务 (15+ 轮) → 监控 prompt token 曲线 → 对比压缩前后
  + 统计文件 I/O 次数 → 验证 L2 缓存命中

路径 C: 恢复正确性
  注入中断 (10 种场景) → Checkpoint 保存 → 恢复 → 对比状态
  → 验证无重复副作用 → 验证无遗漏步骤
```

### 5.3 评测结果

| 维度 | 结果 |
|:-----|:-----|
| **任务成功率** | 100% |
| **预算内完成率** | 100% |
| **Verifier 通过率** | 100% |
| **Prompt 压缩比** | 23% (7,100 → 5,600 tokens) |
| **关键信息丢失** | 0 |
| **重复文件读取** | 60 → 0 次 |
| **中断恢复成功率** | 100% (10/10 场景) |

## 7. 当前局限与展望

### 7.1 已知局限

| 局限 | 说明 |
|:-----|:-----|
| **LLM 后端数量** | 当前仅支持 2 类后端，扩展更多模型需适配工作 |
| **工具类型** | 7 种工具覆盖主流场景，但 LSP 集成深度有限 |
| **Context Budgeting** | 重要性评分依赖启发式规则，未引入语义 embedding |
| **恢复粒度** | 当前以"轮次"为最小恢复单元，无法恢复到单步工具调用级 |
| **并发支持** | 单会话单任务，不支持同一 workspace 的多 Agent 并发 |

### 7.2 演进方向

**短期** (1-3 月)：
- 增加更多 LLM 后端适配（OpenAI-compatible、vLLM 本地部署）
- 工具类型扩展（Docker 沙箱、远程 SSH 执行）
- 恢复粒度下沉到单步工具调用级

**中期** (3-6 月)：
- Context Budgeting 引入语义 embedding 评分
- 多 Agent 并发协作模式
- IDE 插件原生集成（VS Code / JetBrains）

**长期** (6-12 月)：
- 跨会话经验学习（类似 A3 的 ExperienceMemory）
- 自适应安全策略（根据仓库敏感度动态调整审批阈值）
- 分布式 Agent 协作
